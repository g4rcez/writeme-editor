Key Design Decisions

A

Phased Approach (Recommended)

Phase 1: Query parser + engine + Table view + filter builder + bases entity
Phase 2: Kanban view + Calendar view
Phase 3: Autocomplete, syntax highlighting, saved queries

Pros

  • Testable increments
  • Core value delivered fast
  • Views share same engine

Cons

  • Kanban/Calendar delayed
B

All Views at Once

Build all three view types in a single pass. Query engine feeds into a view switcher component that renders table, kanban, or calendar.

Pros

  • Complete feature on delivery
  • Views designed together

Cons

  • Larger single PR
  • Harder to test incrementally
  • Longer time to first result
C

Visual Builder First, Query Language Later

Start with the Notion-style filter UI only. The SQL-like query language comes as a "power mode" in phase 2.

Pros

  • Lower learning curve
  • Simpler initial parser

Cons

  • Power users wait
  • May design filter UI that doesn't map well to SQL later
